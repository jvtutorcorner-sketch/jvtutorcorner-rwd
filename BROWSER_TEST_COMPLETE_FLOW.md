# 完整課程報名到上課測試流程 (手動瀏覽器操作)

本指南提供步驟化的手動瀏覽器操作流程，從學生報名課程到完成上課並驗證點數扣除。

> **操作環境需求**
> - 本地開發環境：`http://localhost:3000`
> - 建議使用 Chrome/Edge 瀏覽器
> - 開啟開發者工具 (F12) 便於觀察 API 請求和設備狀態
> - 建議開啟「Windows 放大鏡」以便同時操作兩個瀏覽器視窗（學生+老師）

---

## 第一步：環境檢查與帳號準備

### 1.1 確認環境變數 (.env.local)

在開始前，確保 `.env.local` 中包含以下配置：

```env
NEXT_PUBLIC_BASE_URL=http://localhost:3000
LOGIN_BYPASS_SECRET=jv_secret_bypass_2024

# 學生帳號
TEST_STUDENT_EMAIL=student@test.com
TEST_STUDENT_PASSWORD=123456

# 老師帳號  
TEST_TEACHER_EMAIL=lin@test.com
TEST_TEACHER_PASSWORD=123456
```

### 1.2 啟動本地開發伺服器

```bash
npm run dev
# 或
yarn dev
```

確認程式成功啟動，訪問 `http://localhost:3000` 能看到首頁。

### 1.3 打開準備操作窗口

為了同時操作學生和老師，建議：
- **方式 1**：開啟 2 個獨立的瀏覽器視窗（或 2 個不同的使用者設檔）
- **方式 2**：使用 Windows 10+ 的分割視窗功能 (`Win + 方向鍵`)
- **方式 3**：使用 Chrome 的多重設檔案功能（隐私模式）

> **為什麼需要 2 個視窗？**  
> 等待頁面需要同時看到學生和老師的狀態同步情況。如只用單視窗切換登入帳號，會失去即時同步的完整驗證。

---

## 第二步：學生報名課程並確認點數扣除

### 2.1 學生視窗 - 登入帳號

1. 在**學生視窗**中訪問 `http://localhost:3000/login`
2. 輸入學生帳號：
   - 帳號：`student@test.com`
   - 密碼：`123456`
3. 點擊「登入」按鈕
4. 系統應自動繞過驗證碼，進入到 `/student_courses` 頁面

### 2.2 檢查初始點數餘額

登入後在首頁或 `/plans` 頁面檢查初始點數。

1. 導航至 `http://localhost:3000/plans`
2. 在頁面右上角或側邊欄看到「點數餘額: XX 點」
3. **記錄初始點數**（例如 100 點）

### 2.3 購買點數（若點數不足）

若點數不足以支付課程（課程通常需要 50-100 點），需要先購買點數：

1. 在 `/plans` 頁面選擇套餐（例如「100 點套餐」）
2. 點擊「購買點數」
3. 進入 `/pricing/checkout` 頁面
4. 選擇「模擬支付」方式（開發環境提供的測試選項）
5. 點擊「確認購買」
6. 系統應返回 `/plans` 頁面，點數餘額應已更新

### 2.4 導航到課程列表並選課

1. 導航至 `http://localhost:3000/courses` 或首頁的課程推薦區
2. 你應該能看到多個課程卡片
3. 選擇其中一個課程（記住課程 ID 和名稱），點擊進入課程詳情頁

### 2.5 執行報名流程

在課程詳情頁：

1. 向下滑動找到「立即報名課程」或「報名」按鈕
2. 點擊該按鈕
3. 系統彈出「選擇付款方式」對話框
4. 選擇「用點數支付」
5. 確認報名（系統應攔截 `/api/orders` 請求，`paymentMethod: 'points'`）
6. 報名成功後，系統應自動跳轉回 `/student_courses` 頁面

**在開發者工具中驗證**：
- 打開 F12 開發者工具 → Network 標籤
- 搜尋 `/api/orders` 的 POST 請求
- 檢查 Request Payload 中確認 `paymentMethod: 'points'`

### 2.6 確認點數扣除

報名成功後，確認點數是否被正確扣除：

1. 導航至 `http://localhost:3000/plans`（或在導覽列點擊「購買點數」）
2. 檢查目前的「點數餘額」
3. **驗算**：`初始點數 - 課程點數成本 = 目前點數`
   - 例如：`100 - 50 = 50 點`（若課程成本 50 點）

**在 API 層驗證**：
- 打開 F12 開發者工具 → Console 標籤
- 執行以下程式碼查詢 API：
  ```javascript
  fetch('http://localhost:3000/api/user/profile')
    .then(res => res.json())
    .then(data => console.log('點數:', data.points))
  ```

## 第三步：學生進入課程清單並驗證時間

### 3.1 檢查「進入教室」按鈕顯示

1. 在學生視窗中訪問 `http://localhost:3000/student_courses`
2. 你應該能看到剛才報名的課程列表
3. 檢查該課程行：
   - **課程名稱**：應正確顯示課程標題
   - **老師**：應顯示老師名稱（非 '-'）
   - **單堂時間**：應顯示分鐘數（例如 "50 m"）
   - **開始時間**：應顯示課程的開始時間（例如 "2026-03-28 14:30:00"）
   - **結束時間**：應顯示課程的結束時間（例如 "2026-03-28 15:20:00"）
   - **進入教室**：若當前時間在開始時間之後，應顯示「進入教室」藍色按鈕

### 3.2 時間驗證邏輯

- **情況 1**：若當前時間在「開始時間 - 10 分鐘」到「結束時間」之間
  - 「進入教室」按鈕應**顯示**
  
- **情況 2**：若當前時間早於「開始時間 - 10 分鐘」或晚於「結束時間」
  - 「進入教室」按鈕應**隱藏**（顯示 "-"）

### 3.3 準備進入教室（學生）

立刻**不要點擊**「進入教室」按鈕，先準備老師端。

---

## 第四步：老師登入並進入課程清單

### 4.1 老師視窗 - 登入帳號

在**老師視窗**中重複以下步驟：

1. 訪問 `http://localhost:3000/login`
2. 輸入老師帳號：
   - 帳號：`lin@test.com`
   - 密碼：`123456`
3. 點擊「登入」按鈕
4. 進入 `/teacher_courses` 頁面

### 4.2 檢查課程列表

在老師的 `/teacher_courses` 頁面：

1. 檢查表格包含以下列：
   - 學生名稱
   - 課程名稱
   - 開始時間
   - 結束時間
   - 單堂時間
   - 剩餘課程數
   - 剩餘時間（分）
   - 進入教室

2. 找到對應的課程（應該能看到剛才報名的學生名稱）
3. 驗證課程資訊：
   - **學生名稱**：應顯示學生的註冊名稱（非 ID）
   - **課程名稱**：應與學生端的課程名稱相同
   - **時間信息**：應與學生端相同
   - **進入教室**：同樣應在指定時間內顯示

## 第五步：同步進入等待頁 (Classroom/Wait)

### 5.1 學生點擊進入教室

**回到學生視窗**，在 `/student_courses` 頁面：

1. 定位到剛才報名的課程
2. 點擊「進入教室」按鈕
3. 系統應導向 `http://localhost:3000/classroom/wait`

### 5.2 等待頁面初始檢查（學生）

在學生的 `/classroom/wait` 頁面，檢查以下元素：

- **課程時間倒數**：應顯示「剩餘 X 分 X 秒」（根據課程時長計算）
- **設備檢測區塊**：
  - 麥克風狀態圖示
  - 攝影機狀態圖示
  - 聲音測試按鈕
- **準備好按鈕**：應為灰色且不可點擊（直到設備檢測完成）
- **進入教室按鈕**：應為灰色且不可點擊（直到老師也準備好）

### 5.3 設備檢測（學生）

學生需要完成設備檢測：

1. 點擊「測試麥克風」按鈕
   - 系統會請求麥克風權限
   - 若出現權限提示，點擊「允許」
   - 應看到麥克風狀態變為「✓ 已就緒」

2. 點擊「測試攝影機」或「測試聲音」
   - 系統會請求攝影機/聲音權限
   - 若出現權限提示，點擊「允許」
   - 應看到攝影機/聲音狀態變為「✓ 已就緒」

3. 設備狀態都顯示「✓ 已就緒」後，「準備好」按鈕應變為藍色而可點擊

### 5.4 老師進入等待頁

**回到老師視窗**，在 `/teacher_courses` 頁面：

1. 定位到對應的課程（應該能看到學生已進入等待頁的狀態提示）
2. 點擊「進入教室」按鈕
3. 系統應導向 `http://localhost:3000/classroom/wait`

### 5.5 設備檢測（老師）

老師重複相同的設備檢測步驟：

1. 點擊「測試麥克風」→ 允許權限 → 等待「✓ 已就緒」
2. 點擊「測試攝影機/聲音」→ 允許權限 → 等待「✓ 已就緒」
3. 設備狀態都就緒後，「準備好」按鈕應變為可點擊

### 5.6 同步檢查

此時，**兩個視窗應都顯示**：

- **學生視窗**：應顯示「老師已準備好」或類似提示
- **老師視窗**：應顯示「學生已準備好」或類似提示

> **如果沒有看到同步狀態更新**  
> - 檢查開發者工具 Network 標籤是否有 SSE 連線 (`/api/classroom/stream`)
> - 或 Polling 請求 (`/api/classroom/ready`)
> - 若 API 報錯，檢查 Console 選項卡的錯誤訊息

---

## 第六步：進入教室並開始上課

### 6.1 學生和老師同時點擊「進入教室」

按照以下順序：

1. **學生視窗**：在等待頁面上點擊「準備好」按鈕
   - 狀態應變為「已準備好」

2. **老師視窗**：在等待頁面上點擊「準備好」按鈕
   - 狀態應變為「已準備好」

3. **等待 2-3 秒**以確保同步

4. **學生視窗**：點擊「進入教室」按鈕
   - 系統應導向 `http://localhost:3000/classroom/room`

5. **老師視窗**：點擊「進入教室」按鈕
   - 系統應導向 `http://localhost:3000/classroom/room`

### 6.2 教室頁面初始檢查

進入教室後（`/classroom/room` 頁面），檢查以下元素：

**視訊 / 音訊區域**：
- 應有 Agora 實時音訊/視訊框
- 左側應顯示自己的視訊或名稱
- 右側應顯示對方的視訊或名稱
- 音訊和視訊控制按鈕應在底部或側邊

**白板區域**：
- 應在主操作區（通常右側或中上方）
- 應有工具列：筆、橡皮擦、文字、圖形等

**課程時間倒數**：
- 應在頁面上方或側邊顯示「剩餘 X 分 X 秒」
- 應開始倒數（每秒 -1）

---

## 第七步：白板操作與隨機畫線

### 7.1 選擇繪筆工具

1. 在白板工具列中選擇「筆」工具
2. 可選：調整筆色和筆粗（若有色板選項）

### 7.2 隨機繪線

在白板上隨機畫幾條線：

1. 點擊白板任意位置開始繪圖
2. 拖動滑鼠畫線
3. 釋放滑鼠鍵結束繪線
4. 重複 3-5 次以繪製多條隨機線

### 7.3 驗證同步

**檢查老師和學生視窗**：

- 學生畫線時，老師視窗應**即時看到**相同的線
- 老師畫線時，學生視窗應**即時看到**相同的線

> 若白板不同步，檢查 Network 標籤中是否有白板相關的 WebSocket 或 API 請求報錯。

## 第八步：結束課程並驗證點數

### 8.1 開始計時（1 分鐘後結束）

1. 記錄當前時間（例如 14:30:00）
2. 頁面上方的「時間倒數」應持續減少
3. **等待 1 分鐘**

### 8.2 老師結束課程

**在老師視窗中**：

1. 尋找「離開教室」或「結束課程」按鈕（通常在右下角或頂部菜單）
2. 點擊該按鈕
3. 系統應彈出確認對話框，例如「確定要結束課程嗎？"
4. 點擊「確認」或「結束」
5. 系統應執行以下操作：
   - 斷開 Agora 連線
   - 呼叫 `/api/enrollments/complete` 或 `/api/orders/complete`（若存在）
   - 更新課程的 `remainingSeconds` 

### 8.3 驗證點數和剩餘課程時間

#### A. 老師端驗證

1. 系統應返回 `/teacher_courses` 頁面
2. 定位到剛才的課程行
3. 檢查「剩餘課程數」和「剩餘時間」欄位
   - 例如：若課程原本是「50 分」，結束課程後應變為「剩餘 XX 分」
   - 應該是 `50 分 - 1 分 = 49 分`（1 分鐘是停留在等待頁的時間，實際上課時間應該更少）

#### B. 學生端驗證

1. **回到學生視窗**
2. 系統應自動返回 `http://localhost:3000/student_courses` 頁面
   - 或手動導航至該頁面
3. 定位到課程行，檢查「剩餘課程數」和「剩餘時間」欄位
   - 應與老師端的數值一致

#### C. API 驗證（開發者工具）

打開 F12 開發者工具 → Network 標籤，尋找以下 API 呼叫：

1. **查詢訂單詳情**
   - 請求：`GET /api/orders?...`
   - 檢查 Response 中的欄位：
     - `remainingSeconds`：應已更新為減少後的值
     - `remainingSessions`：若為多堂課，應減 1 或更新

2. **更新訂單（若適用）**
   - 請求：`PATCH /api/orders/{orderId}`
   - Payload 應包含：`remainingSeconds: XXX`

### 8.4 確認點數狀態

1. 導航至 `http://localhost:3000/plans`
2. 檢查「點數餘額」
3. **由於只是消耗了課程時間，點數不應再變化**
   - 點數應保持報名後的狀態
   - 例如：若報名時扣除 50 點，現在還是 50 點

---

## 第九步：總結驗收清單

完成以上步驟後，檢查以下項目：

- [ ] 學生報名課程成功，點數被正確扣除（初始點數 - 課程成本 = 最終點數）
- [ ] 學生 `/student_courses` 頁面顯示課程資訊完整（無 "-"）
- [ ] 老師 `/teacher_courses` 頁面顯示課程資訊完整且與學生端匹配
- [ ] 進入教室按鈕在時間區間內正確顯示
- [ ] 等待頁面設備檢測三項（麥克風、攝影機、聲音）都能進行
- [ ] 學生與老師的等待狀態能正確同步
- [ ] 教室內白板畫線能即時同步
- [ ] 時間倒數持續減少
- [ ] 老師結束課程後，學生自動返回課程列表
- [ ] **剩餘課程時間在結束課程後正確更新**（若課程只上 1 分鐘，剩餘時間應減 1 分）
- [ ] 點數不再變化（因為點數在報名時已扣除）

## 常見問題排除

### Q1: 「進入教室」按鈕不顯示
**可能原因**：
- 當前時間不在課程時間區間內（檢查系統時間）
- 課程資料未正確返回

**解決方案**：
- 確認系統時間正確
- 若課程是未來時間，可修改系統時間或建立一個當前時間的測試課程

### Q2: 設備檢測卡住或無法請求權限
**可能原因**：
- 瀏覽器未授予麥克風/攝影機權限
- 裝置不支援或未連接設備

**解決方案**：
- 檢查瀏覽器設定 → 隱私 → 網站權限 → 麥克風/攝影機
- 在開發環境中，可绕過硬體檢測（詳見 classroom-wait skill）
- 若用虛擬機，需要啟用虛擬攝影機（例如 OBS Virtual Camera）

### Q3: 等待頁面不同步（看不到對方的準備狀態）
**可能原因**：
- SSE 或 Polling 連線中斷
- API 端點異常

**解決方案**：
- 檢查 F12 Network 標籤的 `/api/classroom/stream` 或 `/api/classroom/ready`
- 刷新頁面重新連線
- 檢查服務器日誌是否有錯誤

### Q4: 白板不同步
**可能原因**：
- WebSocket 連線問題
- 白板 Library 未正確初始化

**解決方案**：
- 檢查 Console 標籤是否有錯誤訊息
- 刷新頁面
- 若問題持續，檢查 `/lib/` 中的白板初始化代碼

### Q5: 結束課程後，剩餘時間未更新
**可能原因**：
- `/api/enrollments/complete` 或 `/api/orders/update` 呼叫失敗
- 前端未正確重新查詢新的剩餘時間

**解決方案**：
- 檢查 F12 Network 標籤中的 API 呼叫是否成功（status 200）
- 手動刷新課程列表頁面，檢查剩餘時間是否已更新
- 若仍未更新，檢查後端 `api/orders` 或 `api/enrollments` 的邏輯

## 進階調試技巧

### 1. 查看實時 API 日誌

在瀏覽器 Console (F12) 執行以下程式碼監控所有 fetch 請求：

```javascript
const originalFetch = window.fetch;
window.fetch = function(...args) {
  console.log('[FETCH]', args[0], args[1]?.body || '');
  return originalFetch(...args);
};
```
