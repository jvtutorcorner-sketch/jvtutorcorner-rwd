# 互動教室「PDF上傳與白板同步」壓力測試報告

**日期：** 2026-06-20  
**報告對象：** 專案驗收單位 / 合作甲方  
**測試範疇：** 互動教室前置流程、`/classroom/wait`（等待室 PDF 上傳）以及 `/classroom/room`（上課教室 PDF 白板同步與斷線重連機制）  
**主要工具：** Playwright 多瀏覽器模擬併發測試框架  

---

## 1. 執行摘要 (Executive Summary)

本測試旨在評估「JV Tutor Corner」互動教室系統在多人同時開課、上傳 PDF 教材以及進行跨端（教師與學生）白板 PDF 頁面同步時的承載上限。

### 核心結論：
* **穩定運作區（1 ~ 3 組併發）：** 在正式環境（Production）中表現極為穩定，所有前置流程、PDF 上傳與即時同步功能皆達到 100% 成功率。
* **臨界觀察區（4 ~ 5 組併發）：** 系統開始出現偶發性的連線逾期。其中 5 組併發時成功率為 80%（4/5），仍達到測試設定的 75% 合格門檻，但已逼近系統承載上限。
* **效能瓶頸區（6 組併發或以上）：** 雖然所有組別的前置註冊、審核、報名與 PDF 上傳均順利完成，但在進入教室後的**白板同步與連線維持階段（Heartbeat）出現失敗**，此時伺服器開始回傳 `500` 錯誤，導致瀏覽器 context 關閉。
* **主要瓶頸點：** 瓶頸並非發生在「前置業務流程」或「PDF 檔案上傳」，而是進入教室後，高併發環境下的 **白板即時通訊服務（Agora Whiteboard Web SDK）連線初始化** 以及 **教室狀態檢測機制（Heartbeat API）**。

---

## 2. 測試環境與設定 (Test Environment & Configuration)

為了真實模擬多組師生同時上課的情境，測試採用自動化腳本，在相同時間內啟動多組獨立的瀏覽器實例（老師端與學生端各一為一組）：

* **測試環境：** 線上正式 Production 環境 (Serverless 雲端架構)
* **單組成員：** 1 位教師 (使用 Chrome 模擬) + 1 位學生 (使用 Chrome 模擬)
* **測試情境設定：**
  1. 教師於後台快速建立課程並經由管理員審核。
  2. 學生扣除點數並完成該課程報名。
  3. 雙方進入 `/classroom/wait` 頁面，教師上傳一個標準 PDF 教案。
  4. 雙方同時進入 `/classroom/room` 互動教室。
  5. 驗證教師端切換 PDF 頁面時，學生端是否能在限時內完成同步。
* **單次課程時長：** 30 分鐘（避免短時間課程失效導致測試誤判）。
* **合格門檻：** 併發組別之白板同步成功率達 **75%** 以上。

## 4. 關鍵發現與問題分析 (Key Findings & Root Cause Analysis)

### 4.1 業務流程具備高度健壯性
測試結果顯示，在 1 到 6 組的壓力下：
* 課程註冊與建立：**100% 成功**
* 管理員審核與授權：**100% 成功**
* 點數扣除與報名流程：**100% 成功**
* PDF 檔案上傳至伺服器：**100% 成功**
這代表系統的「資料庫寫入」、「權限檢驗」、「檔案儲存服務 (S3)」在併發情境下運作完全正常。

### 4.2 失敗點定位於「進入教室後的即時同步階段」
當測試規模提升至 **6 組併發** 時，所有組別雖然都順利進入了 `/classroom/room` 教室，但在等待 PDF 頁面同步驗證（`pdf_sync_verify`）時，所有瀏覽器實例全數失敗，主要錯誤特徵如下：
1. **瀏覽器 Context 異常關閉：**
   出現 `page.waitForFunction: Target page, context or browser has been closed` 錯誤，通常是伺服器未回應、白板 JavaScript 執行緒卡死或瀏覽器因資源耗盡而崩潰。
2. **伺服器 Heartbeat API 回傳 500：**
   瀏覽器內部主控台（Console）錄得複數的伺服器內部錯誤（Status Code: 500），主要集中在互動教室狀態維持與 Heartbeat 更新機制上。

## 5. 測試基礎建設之優化 (Test Infrastructure Improvements)

在執行本次壓力測試過程中，技術團隊同步針對自動化測試環境進行了以下優化，確保測試數據的精準度與可重複性：

1. **精準課程查找策略：** 
   先前測試偶有「學生找不到已報名課程」的狀況，經查是由於歷史測試數據累積，導致自動化腳本誤點了其他名稱相似的過期課程。目前已修正為**優先使用精準 UUID (data-course-id) 進行路由跳轉**，已 100% 排除此雜訊。
2. **靈活的資料復用機制：** 
   新增 `REUSE_STRESS_SETUP` 系列環境變數，允許在本機測試中復用已建立的帳號與課程，避免每次重複執行耗時的前置流程，大幅提升調試效率。
3. **本機錄影權限修正：** 
   修正了 Playwright 錄影 helper 在本機執行時因硬碟磁碟區權限問題導致測試中斷的狀況，統一將路徑預設指向專案內的 `test-results` 目錄。

## 6. 後續改善建議與行動方案 (Recommendations & Actions)

為提升 JV Tutor Corner 互動教室在正式上線後的併發承載力，建議針對以下技術點進行優化：

1. **Heartbeat 頻率與承載優化 (優先級：高)**
   * 目前 Heartbeat API 在高併發時會出現 `500` 錯誤，應對該 Endpoint 進行程式碼審查，確認是否有資料庫連線池（Connection Pool）耗盡或 serverless 逾時問題。
   * 考慮調降 Heartbeat 發送頻率，或將 Heartbeat 機制改為非阻塞式的 Redis / 記憶體暫存。

2. **白板 SDK 初始化優化與錯誤重試 (優先級：高)**
   * 對聲網 Agora Whiteboard（WhiteWebSdk）的初始化邏輯加入更彈性的 Retry 機制。當高併發網路阻塞時，避免直接拋出 Exception 導致教室頁面崩潰。
   * 針對 token 取得與連線建立加入前端 Loading 動態提示與降級方案。

3. **伺服器端請求日誌診斷 (優先級：中)**
   * 安排一輪針對正式環境 Gateway / AP Server 的日誌追蹤，特別定位 6 組併發測試期間的 500 錯誤日誌。

4. **系統容載上限宣告 (建議政策)**
   * 在優化方案部署前，建議將互動教室的單一伺服器/專案建議最大同時上課組數上限宣告為 **3 組（安全）**，至多 **4~5 組（臨界）**。
